**Бета — v0.9.0-beta.1.** DedupCommando выполняет необратимые операции (удаление, hardlink, reflink) над реальными файлами. Прочитайте руководство по безопасности перед применением и держите резервные копии.

DedupCommando — это терминальный инструмент для Linux (CLI и TUI) для поиска **побайтово одинаковых файлов и папок** и освобождения места, которое они занимают впустую. Он рассчитан на хранилища **ZFS**, в том числе размещённые на системах Proxmox VE. Безопасность данных — на первом месте: каждый необратимый пакет выполняется под снимком (snapshot) ZFS, «удалённые» файлы перемещаются в карантин, а не стираются, и содержимое повторно проверяется непосредственно перед каждым действием.

## Безопасность прежде всего

- Перед первым действием создаётся **снимок ZFS** каждого затронутого набора данных (dataset); если хоть один не удался, весь пакет отменяется.
- **«Удаление» перемещает файлы в карантин**, а не вызывает `unlink` — обратимо, пока вы явно его не очистите.
- Содержимое **перепроверяется** (повторный хеш / повторный stat) прямо перед каждым действием; любое расхождение отменяет это действие.
- Файлы публикуются атомарно через `renameat2(RENAME_NOREPLACE)` — без гонки состояний.
- **Блокировка единственного экземпляра** не допускает одновременной записи; перемещения между наборами данных отклоняются — никогда не тихое копирование с последующим удалением.

## Три способа освободить место

- **Удалить в карантин** — уберите дубликат, сохранив возможность восстановить его до очистки.
- **Hardlink** — направьте дубликаты на один inode (в пределах одного набора данных).
- **Reflink** — клонирование блоков с копированием при записи (CoW) на ZFS с `block_cloning` (в одном пуле); независимые метаданные, общие блоки до изменения файла.

В каждой группе один файл — **сохраняемый** (keeper); остальные становятся ссылками или уходят в карантин.

## Как это работает

1. **Сканирование** — обходит выбранные пути, хеширует кандидатов алгоритмом **BLAKE3** (с необязательным побайтовым пересравнением) и группирует одинаковые файлы. Сканирования возобновляемы и кешируются для почти мгновенного повтора.
2. **Просмотр** — изучайте группы дубликатов в многопанельном режиме **commander** (по умолчанию) или в классическом пошаговом мастере (`--classic`); отметьте keeper и действие для каждой группы. Также находятся **«папки-двойники»** — деревья каталогов с одинаковым содержимым.
3. **Применение** — проверьте план и примените его интерактивно или сохраните как shell-скрипт. **Регулятор ресурсов** (Turbo / Balanced / Idle) не даёт сканированию мешать виртуальным машинам и резервным копиям на загруженном хосте.

## Создан для ZFS, работает на Proxmox VE

DedupCommando рассчитан на ZFS: снимки, границы наборов данных и reflink опираются на него. Протестирован на Proxmox VE 9.1 (OpenZFS 2.3), где ZFS доступен «из коробки». Это дедупликация **на уровне файлов** — поиск и удаление одинаковых файлов — а не встроенная блочная дедупликация ZFS (`zfs set dedup`) и не сжатие.

> На файловых системах без ZFS сканирование работает, но защиты снимками нет, поэтому применять действия там не рекомендуется.

## Требования

- **Linux**, ядро ≥ 3.15, x86_64 или aarch64.
- **ZFS настоятельно рекомендуется** (защита снимками, определение наборов данных, reflink); `zfs` в `PATH`, обычно запуск от root.
- Терминал UTF-8 с поддержкой 256 цветов.

## Начало работы

**Debian 13 / Proxmox VE 9+** — установите из подписанного репозитория APT (обновления через `apt upgrade`):

```sh
# as root (Proxmox default); on non-root Debian run: sudo -i
curl -fsSL https://dedupcommando.github.io/apt/dedcom-archive-keyring.gpg \
  -o /usr/share/keyrings/dedcom-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/dedcom-archive-keyring.gpg] https://dedupcommando.github.io/apt stable main" \
  | tee /etc/apt/sources.list.d/dedcom.list
apt update && apt install dedcom
```

Готовые сборки приложены к каждому релизу на GitHub (amd64 и arm64) — скачайте, **проверьте** и установите:

```sh
tar xzf dedcom-<version>-<triple>.tar.gz
install -m 755 dedcom /usr/local/bin/dedcom
```

Сборка из исходников выполняется через Docker, без локального тулчейна Rust.
